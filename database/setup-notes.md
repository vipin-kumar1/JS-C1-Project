# Database Setup Notes

## Choice

SQLite (file-based) via Prisma. Chosen because it is zero-config, requires no external
service, persists to a file so data survives restart, and runs from the README on any
machine with Node installed.

## Where the files live

- Schema (source of truth): `src/backend/prisma/schema.prisma`
- Seed script: `src/backend/prisma/seed.ts`
- Migrations: `src/backend/prisma/migrations/` (generated by `prisma migrate`)
- Database file (git-ignored): `src/backend/prisma/dev.db`
- Environment example: `src/backend/.env.example` (copy to `.env`)

The Prisma files are kept inside the backend package (rather than duplicated here) so the
ORM tooling works out of the box; this file documents them per the required structure.

## Environment

`.env` (copied from `.env.example`):

```
DATABASE_URL="file:./dev.db"
PORT=4000
CORS_ORIGIN="http://localhost:5173"
```

`.env` is git-ignored; only `.env.example` (placeholders) is committed. No secrets.

## Setup steps (local)

```bash
cd src/backend
npm install
cp .env.example .env
npm run prisma:generate     # generate the Prisma client
npm run prisma:migrate      # create the SQLite schema (migration "init")
npm run db:seed             # populate seed data
```

If `prisma migrate dev` cannot auto-run the seed in a restricted shell (an environmental
`tsx` IPC/EPERM error, not a code bug — see `../debugging-notes.md`), run the seed
directly as its own step: `npm run db:seed` (i.e. `npx tsx prisma/seed.ts`).

## Reset

```bash
npm run db:reset            # drops, re-migrates, and re-seeds
```

## Seed contents

- 4 users (1 ADMIN, 3 AGENT).
- 8 tickets spanning every status (OPEN, IN_PROGRESS, RESOLVED, CLOSED, CANCELLED) and
  every priority (LOW, MEDIUM, HIGH), some assigned and one unassigned.
- 4 comments across a few tickets.

## Persistence check

Stop and restart the backend (`npm run dev`); the previously created/updated tickets
remain because they are stored in `dev.db`.
